01 · Context

왜 이 프로젝트가 중요했는가

Club Minish는 기존 고객(초대인)이 지인을 초대하면, 피초대인이 미니쉬 치과에서 상담을 신청하고 방문·시술 후 양측에게 리워드가 지급되는 멤버십 기반 초대 서비스입니다.

서비스 구조 — 두 사용자, 두 개의 화면
초대인 (기존 고객)
앱 (iOS / Android)
① 초대장 구매
② 카카오톡·문자로 링크 전송
⑥ 피초대인 진행 상태 확인
⑦ 리워드 수령
초대 링크 전송
카카오톡 / 문자
피초대인 (신규 사용자)
웹 브라우저 (링크 클릭 → 랜딩)
③ 링크 클릭 → 웹 랜딩페이지 오픈
④ 초대 혜택 확인 + 상담 신청
⑤ 방문 확정 후 시술
⑦ 리워드 수령
피초대인은 앱 설치 없이 링크만으로 웹 브라우저에서 모든 예약 과정을 완료합니다. 이 프로젝트의 핵심 설계 대상은 이 웹 기반 랜딩–예약 흐름이었습니다.

MVP1 단계에서 가장 큰 과제는 초대 → 상담 신청 → 예약 확정 → 방문 확인 → 리워드 지급으로 이어지는 전체 흐름이, 사용자 관점에서는 하나의 경험처럼 느껴지지 않는다는 점이었습니다. 링크를 클릭한 피초대인이 "지금 내가 예약을 한 건지, 신청만 한 건지, 다음에 뭘 해야 하는지"를 명확히 이해하지 못한 채 이탈하는 패턴이 반복됐습니다.

제약 01 — 운영 구조 고려
CS 매니저의 수동 확인 프로세스를 완전히 없애기 어려운 운영 현실 안에서 설계해야 했습니다.
제약 02 — MVP 범위 한정
전체 리뉴얼이 아닌 MVP1 범위 내 개선으로, 기존 기능을 활용하되 인지 부담을 줄이는 방향으로 접근했습니다.
제약 03 — 정보 비대칭
초대인(앱)과 피초대인(웹)이 서로 다른 플랫폼에서 서로 다른 정보를 보는 구조에서, 양쪽 모두에게 명확한 상태 안내가 필요했습니다.
제약 04 — 빠른 런칭 일정
개발 일정을 고려해 구조 변경의 영향 범위를 최소화하면서도 핵심 UX 개선을 반영해야 했습니다.
02 · Research

무엇을 조사했고, 무엇을 발견했는가

조사 방법 규모
피초대 사용자 인터뷰4명
기존 고객 (초대인) 인터뷰2명
CS / 운영 담당자 인터뷰2명
기존 플로우 / 정책 문서 분석—
예약 신청 시나리오 점검—
상태값 / 리워드 정책 매핑—
01
사용자는 혜택보다 "지금 내가 뭘 해야 하는지"를 먼저 알고 싶어 한다. 초대 메시지를 받았을 때 첫 질문은 "예약이 된 건가, 신청만 하는 건가?"였다.
02
신뢰 형성 전 개인정보 입력 요구가 이탈 위험을 높인다. 이름·번호·메모·병원 선택이 초반에 몰리면 사용자는 부담을 느낀다.
03
발행·전송·신청·확정·완료 같은 상태값이 사용자 언어가 아니다. "나는 지금 어디 있는지" 이해하기 어렵다는 피드백이 반복됐다.
04
운영 측면에서 수동 확인 포인트가 5개로, CS 개입이 잦다. 운영 마찰이 결국 사용자의 느린 확정 경험으로 돌아왔다.

04 · 설계 기준

어떤 기준으로 결정했는가

원칙 1
다음 행동이
항상 보여야 한다
현재 상태 표시보다 다음 행동 CTA를 우선 노출한다. 사용자는 "지금 어디"보다 "지금 뭘"을 먼저 알고 싶어 한다.
원칙 2
사용자 언어로
상태를 번역한다
운영 상태값(발행·전송·확정)을 사용자가 직관적으로 이해하는 언어로 재정의한다.
원칙 3
신청과 확정은
명확히 구분한다
'신청 완료'와 '예약 확정'의 혼동을 제거한다. 단계 피드백을 명시적으로 제공해 기대치를 관리한다.
원칙 4
운영 복잡도는
사용자 화면에서 감춘다
내부 확인 프로세스를 사용자에게 노출하지 않는다. 운영 마찰을 백엔드로 이동해 경험을 단순하게 만든다.
문제 1 예약 흐름이 너무 복잡했다

10단계에 달하는 예약 흐름,
사용자는 중간에 멈췄다

초대 링크를 클릭한 피초대인이 상담 신청을 완료하기까지 총 10단계를 거쳐야 했습니다. 별도 인증 화면의 마찰, 지점·날짜·시간·상담분야가 각각 독립된 화면으로 분리된 구조가 이탈의 주요 원인이었습니다.

해결 인지 단계를 재정의했다

기능을 줄인 게 아니라 인지 단계를 재정의했다

10단계를 6단계로 줄인 것의 본질은 화면 수 감소가 아닙니다. 사용자가 경험하는 "결정 포인트"를 재정의하고, 그 사이를 하나의 흐름으로 연결한 것입니다.

Before — 기존 구조 (10단계)
1초대 메시지 수령
2사용자 인증 불필요한 마찰
3초대 정보 + 혜택 확인
4고객 정보 입력
5지점 선택 별도 화면
6날짜 선택 별도 화면
7시간 선택 별도 화면
8상담 분야 선택 별도 화면
9약관 동의
10상담 신청 완료
After — 재설계 구조 (6단계)
1초대 메시지 수령
2초대 정보 + 혜택 확인
3고객 정보 입력 백엔드 자동 인증
4지점 + 날짜 + 시간 + 상담분야 통합 선택
5약관 동의 → 신청 완료 + 단계 안내
인증 화면 제거 — 초대받은 번호와 입력 번호 일치 여부를 백엔드에서 자동 검증. 불일치 시 다음 단계로 진행 불가.

지점 · 날짜 · 시간 · 상담분야
4개 선택 항목이 각각 별도 화면으로 분리되어 있었습니다. 총 4번의 화면 전환이 필요했습니다.
09
약관 동의
이용약관 및 개인정보 수집·이용 동의를 확인합니다.
10
신청 완료
상담 신청이 완료되지만 이후 상태 안내가 없어 사용자가 다음 절차를 알 수 없었습니다.

02
초대 정보 + 혜택 확인
인증 화면 없이 바로 초대 내용과 혜택을 확인합니다.
03 백엔드 자동 인증
고객 정보 입력
입력한 번호와 초대받은 번호가 일치하지 않으면 진행이 차단됩니다. 복잡한 인증 화면은 노출되지 않습니다.

신청 완료
상담신청 → 전화상담 → 방문확정 스텝 표시로 다음에 무슨 일이 일어나는지 즉시 안내해 이탈을 방지합니다.
TO-BE — 초대인 혜택 예약 흐름
초대인 · iOS / Android 앱

초대장을 구매해 지인에게 보낸 초대인은, 피초대인이 방문·시술을 완료한 후 혜택(쿠폰)을 앱에서 직접 사용할 수 있습니다.

웹(피초대인) 플로우를 먼저 실 서비스에 배포해 운영하면서, 단계 축소·정보 통합 등 개선된 예약 설계 원칙이 검증됐습니다. 이후 앱(초대인) 예약 UI를 설계할 때 이 원칙을 그대로 반영해 처음부터 개선된 구조로 디자인했습니다. 별도의 AS-IS 없이 TO-BE만 존재하는 이유입니다.

01
웹 예약 개선 원칙을 앱에 즉시 적용
웹에서 검증된 단계 축소, 정보 통합, 상태 안내 명확화 원칙을 앱 설계에 처음부터 반영. 불필요한 시행착오 없이 바로 개선된 UX로 출발.
02
CS 상담사와의 연결 구조
예약 신청 완료 후 지점 CS가 전화로 예약을 확정하는 구조. "지점에서 전화드려요" 안내를 예약 완료 화면에 명시해 사용자가 다음 단계를 명확히 인지하도록 설계.
03
혜택 유형별 명확한 진입점
쿠폰·초대장 탭 구분으로 혜택 유형별 진입점을 분리. 혜택 선택 → 확인 → 날짜/시간 → 정보 입력 → 완료, 5단계로 간결하게 구성.
01
혜택 선택
앱 내 혜택 탭에서 쿠폰·초대장을 구분해 확인. 유효기간과 제휴 지점 정보가 목록에서 바로 노출됩니다.
02
혜택 확인
쿠폰코드, 유효기간, 사용 가능 지점, 환불 정책을 한 화면에서 확인 후 바로 예약으로 진입합니다.
03
날짜 / 시간 선택
캘린더와 시간 슬롯을 한 화면에 통합. 마감된 시간대는 CS 예약 시스템과 연동해 실시간으로 비활성화되어 처음부터 예약 가능한 시간대만 선택할 수 있습니다.
04
고객 정보 입력 + 개인정보 동의

예약 완료 후에도
사용자는 다음을 알 수 있다

기존에는 "지점에서 전화드려요" 이후 앱에서 예약 상태를 확인할 방법이 없었습니다. 예약신청 → 전화상담 → 예약완료로 이어지는 3단계 상태를 앱에서 실시간으로 표시해, CS가 처리할 때마다 사용자가 즉시 현재 위치를 파악할 수 있도록 개선했습니다.

AS-IS
예약 신청 완료
"지점에서 전화드려요" 이후 앱에서 예약이 어떻게 진행되는지 알 수 없었습니다. 사용자는 전화가 올 때까지 막연히 기다려야 했습니다.
TO-BE 01 — 신청 직후
예약신청 → 전화상담 → 예약완료
신청 완료 즉시 3단계 스텝 바로 현재 위치를 표시합니다. CS 전화상담이 다음 단계임을 바로 인지할 수 있습니다.
TO-BE 02 — 예약 확정 후
CS 처리 완료 → 예약 확정
CS가 확정 처리하면 스텝 바 전체가 완료로 업데이트됩니다. 예약 일시·지점·담당자 정보를 즉시 확인할 수 있습니다.
✓

예약신청 · 전화상담 · 예약완료 3단계를 스텝 바로 표시. CS 처리 시점마다 상태가 업데이트되어 사용자가 현재 단계를 명확히 인지합니다.
✓
예약 정보 즉시 확인
신청 직후부터 지점명·주소·상담 일시·담당자 정보를 앱에서 바로 확인 가능. 확정 후에는 통화하기·주소 복사 CTA로 다음 행동을 안내합니다.
✓
2)·3) 작업 예정
예약 내역 상세 화면(확정 후 정보 표시)과 CS 확정 시 앱 푸시 알림 연동 UI는 추가 작업 후 업데이트 예정입니다.

07 · Reflection

이 프로젝트에서 얻은 것

  • 화면의 단순화만으로는 예약 전환이 개선되지 않는다. 실제 전환의 핵심은 사용자가 현재 상태를 이해하고 다음 행동을 확신하게 만드는 구조 설계였다.
  • 운영자 확인 프로세스는 사용자 경험 바깥의 요소처럼 보이지만, 결국 확정 속도와 메시지 신뢰도에 직접적인 영향을 준다.
  • 단계 수를 줄이는 것보다 각 단계의 역할을 명확히 하는 것이 더 중요했다. "기능 삭제"보다 "인지 단계 재정의"가 핵심이었다.
  • 멤버십 혜택과 예약 흐름을 연결하면 전환이 자연스럽게 높아진다. 혜택은 목적이 아니라 진입 동기여야 한다.

"화면을 줄이는 것이 아니라,
사용자가 생각해야 하는 순간을 없애는 것"
