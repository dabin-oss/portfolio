00 · TL;DR

핵심 요약

문제
초대-상담-예약 흐름이 8단계로 분절돼 사용자가 현재 상태와 다음 액션을 이해하기 어려웠다
접근
사용자 8명 + 운영자 2명 인터뷰와 플로우 분석으로 핵심 구조 문제를 도출하고 재설계를 진행했다
해결
온보딩 단순화, 상태값 언어 재정의, 예약 신청 흐름 통합, 운영 확인 포인트 재구성
결과
프로토타입 기준 과업 완료율과 상태 이해도 개선, 운영 수동 처리 감소
01 · Context

왜 이 프로젝트가 중요했는가

Club Minish는 기존 고객(초대인)이 지인을 초대하면, 피초대인이 미니쉬 치과에서 상담을 신청하고 방문·시술 후 양측에게 리워드가 지급되는 멤버십 기반 초대 서비스입니다.

MVP1 단계에서 가장 큰 과제는 초대 → 상담 신청 → 예약 확정 → 방문 확인 → 리워드 지급으로 이어지는 전체 흐름이, 사용자 관점에서는 하나의 경험처럼 느껴지지 않는다는 점이었습니다.

제약 01 — 운영 구조 고려
CS 매니저의 수동 확인 프로세스를 완전히 없애기 어려운 운영 현실 안에서 설계해야 했습니다.
제약 02 — MVP 범위 한정
전체 리뉴얼이 아닌 MVP1 범위 내 개선으로, 기존 기능을 활용하되 인지 부담을 줄이는 방향으로 접근했습니다.
제약 03 — 정보 비대칭
초대인과 피초대인이 서로 다른 정보를 보는 구조에서, 양쪽 모두에게 명확한 상태 안내가 필요했습니다.
제약 04 — 빠른 런칭 일정
개발 일정을 고려해 구조 변경의 영향 범위를 최소화하면서도 핵심 UX 개선을 반영해야 했습니다.
02 · Research

무엇을 조사했고, 무엇을 발견했는가

조사 방법 규모
피초대 사용자 인터뷰4명
기존 고객 (초대인) 인터뷰2명
CS / 운영 담당자 인터뷰2명
기존 플로우 / 정책 문서 분석—
예약 신청 시나리오 점검—
상태값 / 리워드 정책 매핑—
01
사용자는 혜택보다 "지금 내가 뭘 해야 하는지"를 먼저 알고 싶어 한다. 초대 메시지를 받았을 때 첫 질문은 "예약이 된 건가, 신청만 하는 건가?"였다.
02
신뢰 형성 전 개인정보 입력 요구가 이탈 위험을 높인다. 이름·번호·메모·병원 선택이 초반에 몰리면 사용자는 부담을 느낀다.
03
발행·전송·신청·확정·완료 같은 상태값이 사용자 언어가 아니다. "나는 지금 어디 있는지" 이해하기 어렵다는 피드백이 반복됐다.
04
운영 측면에서 수동 확인 포인트가 5개로, CS 개입이 잦다. 운영 마찰이 결국 사용자의 느린 확정 경험으로 돌아왔다.

04 · 설계 기준

어떤 기준으로 결정했는가

Principle 01
다음 행동이 항상 보여야 한다
현재 상태 표시보다 다음 행동 CTA를 우선 노출한다. 사용자는 "지금 어디"보다 "지금 뭘"을 먼저 알고 싶어 한다.
Principle 02
사용자 언어로 상태를 번역한다
운영 상태값(발행·전송·확정)을 사용자가 직관적으로 이해하는 언어로 재정의한다.
Principle 03
신청과 확정은 명확히 구분한다
'신청 완료'와 '예약 확정'의 혼동을 제거한다. 단계 피드백을 명시적으로 제공해 기대치를 관리한다.
Principle 04
운영 복잡도는 사용자 화면에서 감춘다
내부 확인 프로세스를 사용자에게 노출하지 않는다. 운영 마찰을 백엔드로 이동해 경험을 단순하게 만든다.
05 · Solution

기능을 줄인 게 아니라 인지 단계를 재정의했다

8단계를 4단계로 줄인 것의 본질은 화면 수 감소가 아닙니다. 사용자가 경험하는 "결정 포인트"를 4개로 재정의하고, 그 사이를 하나의 흐름으로 연결한 것입니다.

Before — 기존 구조 (8단계)
1초대 메시지 수령
2사용자 인증
3초대 정보 + 혜택 확인 (정보 과다)
4고객 정보 입력
5지점 선택 (별도 화면)
6날짜 선택 (별도 화면)
7시간 선택 (별도 화면)
8신청 완료 (상태 불명확)
After — 재설계 구조 (4단계)
1초대 이해 (서비스 소개 통합)
2상담 신청 (지점·일정 통합)
3정보 확인 + 혜택 선택
4신청 완료 + 단계 안내
Key Screen 01 · 정보 확인 + 혜택 통합
초대 확인 화면
혜택 상세
  • Before: 정보 확인 화면 → 뒤로 → 혜택 선택 → 다시 앞으로 반복
  • After: 상담 정보 + 혜택 선택 + 약관이 한 화면에 통합
  • After: 치아 선물 사용 여부도 즉시 확인·선택 가능
Key Screen 02 · 단계 피드백
신청 완료
방문 확정
  • Before: 신청 후 다음 절차가 불명확 → 이탈 위험
  • After: 상담신청 → 전화상담 → 방문확정 스텝 바로 현재 위치 표시
  • After: 지점명·주소·상담 정보·초대장 정보 한눈에 요약
Key Screen 03 · 혜택 홈 재구성
혜택 홈

혜택과 예약이 분리된 기존 구조에서, 혜택 홈에서 포인트·일정·초대장을 한눈에 보고 바로 상담 신청으로 이어지도록 CTA 흐름을 재설계했습니다.

  • Before: 혜택 확인 후 별도 예약 메뉴를 다시 찾아야 했음
  • After: 포인트 · 내 일정 · 초대장이 혜택 홈에 통합
  • After: 초대장에서 바로 상담신청 CTA로 연결

07 · Reflection

이 프로젝트에서 얻은 것

  • 화면의 단순화만으로는 예약 전환이 개선되지 않는다. 실제 전환의 핵심은 사용자가 현재 상태를 이해하고 다음 행동을 확신하게 만드는 구조 설계였다.
  • 운영자 확인 프로세스는 사용자 경험 바깥의 요소처럼 보이지만, 결국 확정 속도와 메시지 신뢰도에 직접적인 영향을 준다.
  • 단계 수를 줄이는 것보다 각 단계의 역할을 명확히 하는 것이 더 중요했다. "기능 삭제"보다 "인지 단계 재정의"가 핵심이었다.
  • 멤버십 혜택과 예약 흐름을 연결하면 전환이 자연스럽게 높아진다. 혜택은 목적이 아니라 진입 동기여야 한다.

"화면을 줄이는 것이 아니라,
사용자가 생각해야 하는 순간을 없애는 것"
