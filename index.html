00 TL;DR — 핵심 요약
문제
초대-상담-예약 흐름이 8단계로 분절돼 사용자가 현재 상태와 다음 액션을 이해하기 어려웠다
접근
사용자 8명 + 운영자 2명 인터뷰와 플로우 분석으로 핵심 구조 문제를 도출하고 재설계를 진행했다
해결
온보딩 단순화, 상태값 언어 재정의, 예약 신청 흐름 통합, 운영 확인 포인트 재구성
결과
프로토타입 기준 과업 완료율과 상태 이해도 개선, 운영 수동 처리 감소
01 Context — 배경과 제약

왜 이 프로젝트가
중요했는가

Club Minish는 기존 고객(초대인)이 지인을 초대하면, 피초대인이 미니쉬 치과에서 상담을 신청하고 방문·시술 후 양측에게 리워드가 지급되는 멤버십 기반 초대 서비스입니다.

MVP1 단계에서 가장 큰 과제는 초대 → 상담 신청 → 예약 확정 → 방문 확인 → 리워드 지급으로 이어지는 전체 흐름이, 사용자 관점에서는 하나의 경험처럼 느껴지지 않는다는 점이었습니다.

특히 처음 서비스를 접하는 피초대인은 "나는 지금 어디 있는가" "예약이 된 건가, 신청만 한 건가"를 구분하지 못한 채 이탈하는 경우가 많았습니다.

단순한 화면 개선이 아니라, 서비스 흐름 자체의 재구조화가 필요한 프로젝트였습니다.

제약 01 — 운영 구조 고려CS 매니저의 수동 확인 프로세스를 완전히 없애기 어려운 운영 현실 안에서 설계해야 했습니다.
제약 02 — MVP 범위 한정전체 리뉴얼이 아닌 MVP1 범위 내 개선으로, 기존 기능을 활용하되 인지 부담을 줄이는 방향으로 접근했습니다.
제약 03 — 정보 비대칭초대인과 피초대인이 서로 다른 정보를 보는 구조에서, 양쪽 모두에게 명확한 상태 안내가 필요했습니다.
제약 04 — 빠른 런칭 일정개발 일정을 고려해 구조 변경의 영향 범위를 최소화하면서도 핵심 UX 개선을 반영해야 했습니다.
02 Research — 근거와 발견

무엇을 조사했고,
무엇을 발견했는가

포트폴리오용 재구성 데이터입니다. 실제 서비스 운영 관찰 및 시나리오 분석을 기반으로 구성했으며, 인터뷰는 유사 상황의 사용자와 진행한 내용을 재정리했습니다.

핵심 인사이트
Insight 01
사용자는 혜택보다 "지금 내가 뭘 해야 하는지"를 먼저 알고 싶어 한다. 초대 메시지를 받았을 때 첫 질문은 "이게 뭔 서비스지?"가 아니라 "예약이 된 건가, 신청만 하는 건가?"였다.
Insight 02
신뢰 형성 전 개인정보 입력 요구가 이탈 위험을 높인다. 이름·번호·메모·병원 선택이 초반에 몰리면 사용자는 부담을 느낀다.
Insight 03
발행·전송·신청·확정·완료 같은 상태값이 사용자 언어가 아니다. "나는 지금 어디 있는지" 이해하기 어렵다는 피드백이 반복됐다.
Insight 04
운영 측면에서 수동 확인 포인트가 5개로, CS 개입이 잦다. 운영 마찰이 결국 사용자의 느린 확정 경험으로 돌아왔다.

01
상태 이해
사용자는 현재 상태와 다음 액션을 이해하기 어렵다
발행·전송·신청·확정·완료의 상태값은 운영 언어이지, 사용자 언어가 아니다. "나는 지금 어디 있는가"가 보이지 않는다.
02
흐름 단절
예약 경험이 기능 단위로 분리돼 완료 흐름이 끊긴다
초대·정보 입력·일정 선택·확인이 각각 분절되어 인지 부담이 크다. 사용자는 "다음에 뭘 해야 하지?"를 반복해서 묻게 된다.
03
운영 마찰
운영 프로세스가 사용자 경험에 숨은 마찰을 만든다
CS 수동 확인이 필요한 구조는 결국 사용자에게 느린 확정 경험으로 돌아온다. 운영 복잡도가 화면 밖에서 작동한다.
05 Design Principles — 판단 기준

어떤 기준으로
결정했는가

Principle 01
다음 행동이 항상 보여야 한다
현재 상태 표시보다 다음 행동 CTA를 우선 노출한다. 사용자는 "지금 어디"보다 "지금 뭘"을 먼저 알고 싶어 한다.
Principle 02
사용자 언어로 상태를 번역한다
운영 상태값(발행·전송·확정)을 사용자가 직관적으로 이해하는 언어로 재정의한다.
Principle 03
신청과 확정은 명확히 구분한다
'신청 완료'와 '예약 확정'의 혼동을 제거한다. 단계 피드백을 명시적으로 제공해 기대치를 관리한다.
Principle 04
운영 복잡도는 사용자 화면에서 감춘다
내부 확인 프로세스를 사용자에게 노출하지 않는다. 운영 마찰을 백엔드로 이동해 경험을 단순하게 만든다.

08 Reflection — 배운 것

이 프로젝트에서
얻은 것

배운 점
화면의 단순화만으로는 예약 전환이 개선되지 않는다. 실제 전환의 핵심은 사용자가 현재 상태를 이해하고 다음 행동을 확신하게 만드는 구조 설계였다.
운영자 확인 프로세스는 사용자 경험 바깥의 요소처럼 보이지만, 결국 확정 속도와 메시지 신뢰도에 직접적인 영향을 준다.
단계 수를 줄이는 것보다 각 단계의 역할을 명확히 하는 것이 더 중요했다. "기능 삭제"보다 "인지 단계 재정의"가 핵심이었다.
멤버십 혜택과 예약 흐름을 연결하면 전환이 자연스럽게 높아진다. 혜택은 목적이 아니라 진입 동기여야 한다.
